## Root Cause Analysis

### Bug 1: Airways 91% Data Loss

**Location**: `adl/scripts/airac_update.py` lines 280-306

**Mechanism**:
1. `dbo.airways` has a UNIQUE index `IX_airway_name` on `airway_name` column (verified on both VATSIM_REF and VATSIM_ADL)
2. The CSV contains 16,988 airway entries with only 10,287 unique names — the remaining 6,701 are legitimate multi-region variants (e.g., airway W3 exists as 19 separate geographic segments across different countries)
3. The import uses a single `cursor.executemany()` for ALL 16,988 rows (line 298)
4. When the first duplicate name is encountered, the entire batch fails and rolls back
5. Only ~1,515 rows survive from a partial pre-duplicate section

**Evidence**:
- Top duplicated airways verified from CSV: W6 (19 instances), W3 (19), W19 (19), W8 (18), W4 (17), V3 (17)
- Each instance has a completely different `fix_sequence` (different geographic regions)
- PostGIS has all 16,988 because it was loaded directly from CSV without the UNIQUE constraint

### Bug 2: Airway Segments 100% Loss

**Location**: `adl/scripts/airac_update.py` line 277

**Mechanism**:
1. `TRUNCATE TABLE dbo.airway_segments` clears all segments (line 277)
2. No code exists in the Python pipeline to regenerate segments
3. The only segment generator is `adl/reference_data/import_airways.php` (PHP), which is not called by the Python pipeline
4. The PHP script uses a sophisticated 3-pass bidirectional fix disambiguation algorithm with equirectangular proximity resolution

**Evidence**:
- VATSIM_REF: 0 rows, VATSIM_ADL: 0 rows
- PostGIS: 91,808 rows (from PHP pipeline or direct import)
- `airway_segments` table schema exists with 15 columns including `segment_geo` geography

### Bug 3: All `_old_` Historical Entries Discarded

**Location**: `adl/scripts/airac_update.py` lines 118, 142, 240, 349, 452

**Mechanism**:
- Every import function contains `if '_old_' in name.lower(): continue`
- This silently drops all entries with `_old_` suffix from the CSV
- The `_old_` entries represent previous AIRAC versions of modified/moved/removed navigation data
- No database storage exists for this historical information

**Scale of discarded data** (verified from CSV files):
- points.csv: 2,771 `_old_` entries
- navaids.csv: 114 `_old_` entries
- awys.csv: 78 `_old_` entries
- cdrs.csv: 6,003 `_old_` entries
- dp_full_routes.csv: 2,063 `_old_` entries
- star_full_routes.csv: 2,352 `_old_` entries
- playbook_routes.csv: 28,441 `_old_` entries (51% of all playbook data!)

### Bug 4: PostGIS Sync Incomplete

**Location**: `scripts/postgis/sync_ref_to_postgis.py` line 322

**Mechanism**:
- The sync function only handles 3 of 7 tables: `nav_fixes`, `airways`, `area_centers`
- Missing: `coded_departure_routes`, `nav_procedures`, `playbook_routes`
- The `airway_segments` sync exists in code but syncs 0 rows (since REF has 0)

### Additional Issue: No Changelog Tracking

- No `navdata_changelogs` table exists in any database
- No mechanism to track what changed between AIRAC cycles
- The `nasr_navdata_updater.py` script DOES generate local changelog files (JSON + Markdown in `assets/data/logs/`), but this data is never persisted to the database

## `_old_` Entry Format Documentation

### Verified Naming Conventions (from actual CSV data)

Four formats exist:

**Modern format**: `{base_name}_old_{YYNN}_{reason}`
- `AANAH_old_2602_moved` — waypoint coordinates changed >0.0001 degrees (~11m)
- `V1_old_2602_changed` — airway fix sequence modified
- `ACE_old_2602_removed` — entry removed from FAA NASR source
- `TEEKY3_old_2602_superseded` — procedure amendment bumped (e.g., TEEKY3 → TEEKY4)

**Legacy pre-format**: `{base_name}_old_pre{YYNN}`
- `AA_old_pre2602` — pre-cycle-2602 version (no specific reason recorded)
- `G8_old_pre2602` — airway version before cycle 2602

**Cycle-only format**: `{base_name}_old_{YYNN}`
- `PLAY_old_2601` — playbook style, cycle only, no reason suffix

**Bare legacy format**: `{base_name}_OLD`
- `NAME_OLD` — uppercase, no cycle or reason metadata

**Reasons vocabulary**: `moved`, `changed`, `removed` (observed in current CSVs); `superseded` (supported in schema, from `nasr_navdata_updater.py` procedure amendment detection, not yet present in current data)

### Change Detection Thresholds (from `nasr_navdata_updater.py`)
- **moved**: coordinate diff > 0.0001 degrees (~11 meters ground distance)
- **changed**: non-coordinate content modified (fix_sequence, full_route, etc.)
- **superseded**: procedure amendment number changed (base code same, version number different)
- **removed**: entry present in previous AIRAC cycle but absent in new cycle

## International Data Coverage Analysis

### XP12/Navigraph Integration (from `nasr_navdata_updater.py`)

The NASR updater merges data from two sources:
1. **FAA NASR** (primary): US domestic fixes, airways, procedures
2. **X-Plane 12 Navigraph** (supplemental): International data from `earth_fix.dat`, `earth_nav.dat`, `earth_awy.dat`

Merge rules:
- XP12 data is merged AFTER NASR data
- New unique names from XP12 are added directly
- If both sources have the same airway name, the version with more fixes wins (NASR takes precedence if equal)
- Multi-region variants (disconnected geographic segments) are detected via connected-component analysis

### Verified International Coverage in CSVs

**points.csv**: 265,322 non-US waypoints (69.4% of total)
- Canadian waypoints: YAABA, YAAKK, YBC, YVR, YYC, etc.
- Pacific: AADPO (-14.17, -170.97), ACRON (13.62, 143.03)
- European, Asian, African waypoints all present

**awys.csv**: International airway prefixes found:
- A-series (A1, A315): International standard
- B-series (B1, B3, B932): International
- G-series (G1-G8): International
- L-series (L1-L9): Low-altitude international
- M-series: Military/mixed
- N-series: North Atlantic
- R-series: International
- UB/UG/UL/UR: Upper-level international
- W-series (W1-W976): Asia-Pacific regional (most duplicated names; 1,969 entries)

**Character length analysis**:
- All active airway names: ≤8 characters (fits current nvarchar(8))
- Active fix names: 353 fixes exceed 16 characters (e.g., `ZZ_RJDG_SOUTH_KYUSHU_HIGH_01` = 28 chars). These are `ZZ_` pseudo-fixes for international FIR/sector regions. 3 truncation collisions at 16 chars. **Fix**: widen `fix_name` to `nvarchar(32)`.
- `_old_`-suffixed names exceed column limits (e.g., `B932_old_2602_changed` = 22 chars) — handled by stripping suffix before storage

## nav_procedures PostGIS Gap Explanation

PostGIS has 97,889 rows vs VATSIM_REF's 10,314. This is **NOT a bug** — it's expected.

**Breakdown**:
- ~10,314 rows from NASR CSVs (dp_full_routes.csv + star_full_routes.csv minus _old_ entries)
- ~87,575 rows from CIFP base transitions generated by `scripts/postgis/generate_base_transitions.py`

The `adl/reference_data/generate_base_transitions.py` script:
1. Reads CIFP `.dat` files from X-Plane/Navigraph (`C:\X-Plane 12\Custom Data\CIFP`)
2. Parses route_type=5 (enroute/common portion) for STARs and DPs
3. Generates synthetic "base transition" records with `source='cifp_base'`
4. Writes to SQL Server `dbo.nav_procedures` (VATSIM_ADL); CIFP records reach PostGIS via a separate sync path

This CIFP data is richer and more granular than the CSV-based import. The PostGIS sync strategy for nav_procedures must preserve this CIFP data.

### 2a. Add `parse_old_suffix()` helper

```python
import re

def parse_old_suffix(name: str) -> tuple:
    """Parse _old_ suffix from AIRAC versioning.

    Returns (clean_name, is_superseded, cycle, reason).

    Formats handled:
      MERIT_old_2602_moved     -> ('MERIT', True, '2602', 'moved')
      AA_old_pre2602           -> ('AA', True, 'pre2602', None)
      PLAY_old_2601            -> ('PLAY', True, '2601', None)
      V1_old_2602_changed      -> ('V1', True, '2602', 'changed')
      NAME_OLD                 -> ('NAME', True, None, None)
      NORMAL_NAME              -> ('NORMAL_NAME', False, None, None)
    """
    # Full format: name_old_YYNN_reason
    match = re.match(r'^(.+?)_old_(\d{4})_(\w+)$', name, re.IGNORECASE)
    if match:
        return match.group(1), True, match.group(2), match.group(3)

    # Legacy format: name_old_preYYNN
    match = re.match(r'^(.+?)_old_(pre\d{4})$', name, re.IGNORECASE)
    if match:
        return match.group(1), True, match.group(2), None

    # Catch-all: name_old_anything (includes cycle-only like _old_2601)
    match = re.match(r'^(.+?)_old_(.+)$', name, re.IGNORECASE)
    if match:
        return match.group(1), True, match.group(2), None

    # Bare legacy: name_OLD (no underscore-delimited suffix)
    match = re.match(r'^(.+?)_OLD$', name)
    if match:
        return match.group(1), True, None, None

    return name, False, None, None
```

### 2d. Add airway segment generation

New function `generate_airway_segments(conn)` called after airways import.

**Algorithm** (equirectangular proximity disambiguation):
1. Load all `nav_fixes` into dict: `{fix_name: [(lat, lon), ...]}` (handles multi-instance names)
2. For each non-superseded airway, split `fix_sequence` into fix names
3. Resolve each fix to coordinates:
   - **First fix**: Look-ahead — find first fix with only 1 instance as anchor. If none, try closest-pair of first two ambiguous fixes.
   - **Subsequent fixes**: Pick instance closest (equirectangular distance) to previously resolved fix.
4. For each consecutive pair, compute:
   - `distance_nm` via haversine
   - `course_deg` via initial bearing
5. **Validation**: Skip segments where `distance_nm > 1000` (bad disambiguation). Log count.
6. Batch insert to `dbo.airway_segments` (500-row batches, per-row retry)

**Helper functions**:
```python
import math

def haversine_nm(lat1, lon1, lat2, lon2):
    R = 3440.065  # Earth radius in nm
    dlat = math.radians(lat2 - lat1)
    dlon = math.radians(lon2 - lon1)
    a = (math.sin(dlat/2)**2 +
         math.cos(math.radians(lat1)) * math.cos(math.radians(lat2)) *
         math.sin(dlon/2)**2)
    return R * 2 * math.asin(min(1.0, math.sqrt(a)))

def initial_bearing_deg(lat1, lon1, lat2, lon2):
    lat1r, lon1r = math.radians(lat1), math.radians(lon1)
    lat2r, lon2r = math.radians(lat2), math.radians(lon2)
    dlon = lon2r - lon1r
    x = math.sin(dlon) * math.cos(lat2r)
    y = (math.cos(lat1r) * math.sin(lat2r) -
         math.sin(lat1r) * math.cos(lat2r) * math.cos(dlon))
    return (math.degrees(math.atan2(x, y)) + 360) % 360

def equirect_dist_sq(lat1, lon1, lat2, lon2):
    dlat = lat2 - lat1
    dlon = (lon2 - lon1) * math.cos(math.radians((lat1 + lat2) / 2))
    return dlat * dlat + dlon * dlon
```

### Corrections Required (with Resolutions)

**1. Airways `_old_` filtering bug is larger than documented.**
- Current code truncates airway name to 8 chars (`[:8]`, line 239) before `_old_` check (line 240).
- 140 historical airway rows exist; only 32 caught after truncation; 108 slip through.
- **Resolution**: Our fix removes both the `[:8]` truncation AND the `_old_` filter. With `nvarchar(30)` columns and `parse_old_suffix()`, all _old_ entries are preserved with stripped base names and supersession metadata. No leakage possible.

**2. `_old_` format handling needs additional real formats.**
- Observed: `name_old_2602_changed`, `name_old_pre2602`, `name_OLD` (legacy), `name_old_2601` (cycle only, no reason).
- **Resolution**: Updated `parse_old_suffix()` to handle all 4 patterns:
  ```python
  # Full: MERIT_old_2602_moved -> ('MERIT', True, '2602', 'moved')
  # Legacy pre: AA_old_pre2602 -> ('AA', True, 'pre2602', None)
  # Cycle only: PLAY_old_2601 -> ('PLAY', True, '2601', None)
  # Bare legacy: NAME_OLD -> ('NAME', True, None, None)
  ```

**3. `superseded` reason not present in current CSVs.**
- Only `moved`, `changed`, `removed` observed in current data.
- **Resolution**: `superseded` kept as valid future reason in schema (from `nasr_navdata_updater.py` procedure amendment detection). No action needed.

**4. Migration path/versioning conflicts.**
- `database/migrations/navdata/` does not exist.
- `database/migrations/postgis/010_supersession_columns.sql` conflicts with existing `010_pseudo_fix_skip.sql`.
- **Resolution**: Use `database/migrations/schema/` (existing family) for Azure SQL migration. PostGIS migration renumbered to `011_supersession_columns.sql`.

**5. CIFP/base-transition path and DB-target.**
- Script is `adl/reference_data/generate_base_transitions.py` (not `scripts/postgis/`).
- Writes to SQL Server `dbo.nav_procedures` (VATSIM_ADL), not PostGIS.
- CIFP records reach PostGIS via a separate sync path, not from VATSIM_REF.
- **Resolution**: Plan text corrected. PostGIS nav_procedures UPSERT strategy remains correct — delete `source='NASR'` records only, preserving CIFP-sourced records regardless of how they arrived.

**6. Fix names exceed 16 chars.**
- 353 fixes > 16 chars in CSV: `ZZ_` pseudo-fixes (e.g., `ZZ_RJDG_SOUTH_KYUSHU_HIGH_01` = 28 chars) and `_old_` suffixed names.
- 3 truncation collisions (all `ZZ_` pseudo-fixes, not real navaids).
- **Resolution**: Widen `fix_name` from `nvarchar(16)` to `nvarchar(32)` in both VATSIM_REF and VATSIM_ADL. Remove `[:16]` truncation from importer. This preserves full international pseudo-fix names and eliminates collisions.

**7. W-series extends to W976.**
- 1,969 W-series entries across W1-W976.
- **Resolution**: Plan text corrected. No code change needed — airway type classification already handles W-prefix.

**8. Phase 3 area_centers supersession scope.**
- `area_centers` is not in the 5-table supersession scope.
- **Resolution**: Phase 3 will NOT add supersession columns to `sync_area_centers()`. Only `sync_nav_fixes()`, `sync_airways()`, `sync_coded_departure_routes()`, `sync_nav_procedures()`, `sync_playbook_routes()` get supersession columns.
